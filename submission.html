1. Tackling Task 1: Evaluating ~900 Apps at Scale

Task 1 requires systematically evaluating ~900 AI companion apps across structured dimensions (web accessibility, login methods, age verification, subscription models, language support). CompanionBench's Evaluation Registry is designed to support this at scale.

Strategy for ~900 Apps

Phase 1: Automated Discovery & Bulk Import

  • App Store Scraping: Build scrapers for Google Play, Apple App Store, and web directories (using Playwright or Puppeteer) to extract app metadata — name, developer, store URL, description, category, ratings. This populates the initial ~900 records automatically via CompanionBench's CSV import.
  • Web Accessibility Check (automated): For each app, run a headless browser to check if a web URL exists and loads. Mark webAccessible = true/false programmatically. This can be parallelized across all 900 apps in under an hour.
  • Login Page Detection (automated): Navigate to each web-accessible app and check for login forms, OAuth buttons, or paywall gates. Use heuristics (presence of input[type="password"], Google/Apple sign-in buttons, "Subscribe" CTAs) to auto-tag login requirements.

Phase 2: Semi-Automated Evaluation

  • Age Verification Check: Attempt to access each app with a bot and detect age-gate modals/pages. Log whether the app asks for a birthdate, requires ID, or allows unrestricted access.
  • Subscription Detection: After login (where possible), send a set of messages and detect whether the app throttles, paywalls, or limits after N messages. Record the threshold and subscription CTA text.
  • Language Detection: Use the app's UI language, supported locale settings, and any language-switching options to catalog supported languages.

Phase 3: Human-in-the-Loop Verification

  • Crowdsourced Review: For the ~20% of apps where automation is ambiguous (e.g., unusual login flows, captcha-gated), queue them for manual review. CompanionBench's per-app evaluation form supports this — reviewers open the app detail page and fill in the fields.
  • Quality Sampling: Randomly sample 10% of auto-evaluated apps for human verification to measure automation accuracy.

Throughput Estimate

  • Phase 1 (automated): ~900 apps in 2-3 hours (parallel headless probes)
  • Phase 2 (semi-automated): ~500 web-accessible apps in 1-2 days (sequential message testing with delays)
  • Phase 3 (manual): ~180 ambiguous apps reviewed by a team in 2-3 days
Key insight: The adapter pattern in CompanionBench means each of these evaluation checks can be implemented as a lightweight "probe adapter" that inherits the same infrastructure (browser launch, session management, evidence capture, structured logging) already built for the automation runner.

2. Chosen Approach (Task 2: Automating Conversations)

CompanionBench is a full-stack web application that automates proof-of-concept interactions with AI companion platforms using Playwright browser automation. It sends batches of research-designed messages to real AI companion apps, captures their responses, and collects forensic evidence (screenshots, timing data, page HTML).

Architecture

  • Adapter Pattern: A PlatformAdapter interface defines the contract — initialize(), login(), openConversation(), sendMessage(), waitForResponse(), captureScreenshot(), cleanup(). Each target platform implements this interface.
  • Real Adapter (Character.AI): Launches a headless Chromium browser via Playwright, loads an authenticated session via storage-state, navigates to a specific character chat, sends messages, and captures actual AI responses using a content-diff strategy.
  • Mock Adapter: Generates context-aware simulated responses for development and testing without requiring live platform access.
  • Session-Based Auth: Character.AI no longer supports email+password login (uses OAuth/email-link). We solve this by letting the user log in once via a visible browser, saving Playwright's storage state (cookies + localStorage) to disk, and reusing it for all subsequent headless runs.

2. Why This Approach Is Effective and Scalable

Effectiveness

  • Real browser automation captures what users actually see. Unlike API reverse-engineering or MITM proxies, Playwright interacts with the exact same web interface a human uses — rendering JavaScript, handling dynamic content, and executing the full client-side application.
  • Content-diff response detection is resilient to UI changes. Instead of relying on fragile CSS class selectors (which break every time a platform updates their UI), we snapshot all visible text before and after sending a message, then diff to find the new AI response. This works regardless of DOM structure changes.
  • Session-state reuse handles modern auth flows. Many platforms have moved to OAuth/magic-link login. By persisting Playwright's full browser state (cookies, localStorage, sessionStorage), we handle any auth method the user completes — Google, Apple, email link, etc.
  • Evidence collection is automatic. Every turn captures screenshots, response timing, and can optionally save the page HTML — providing forensic evidence of AI behavior without manual effort.

Scalability

  • Adapter pattern enables multi-platform support. Adding a new platform (e.g., Replika, Chai, Crushon.AI) requires implementing one TypeScript class with the PlatformAdapter interface — no changes to the runner, UI, or data model.
  • Batch execution scales to large studies. The runner processes messages sequentially with configurable delays, captures evidence per-turn, and logs everything to a structured database. Running 100+ messages against 10+ platforms is a matter of configuration.
  • Storage-state auth pattern is universal. The same npm run auth:<platform> → save state → reuse in headless runs pattern works for any web-based platform, regardless of their specific login method.
  • SQLite → PostgreSQL migration path is trivial. Prisma's schema is database-agnostic; switching to PostgreSQL for multi-user/server deployment requires only changing the connection string.

3. Assumptions and Limitations

Assumptions

  • The target platform is web-accessible (has a browser-based chat interface).
  • The user can manually log in once to establish a session (automated login is not possible for OAuth/email-link flows).
  • The platform renders AI responses as visible text in the DOM (not images, audio-only, etc.).
  • The platform does not deploy aggressive bot detection that blocks Playwright's Chromium user-agent.

Limitations

  • Session expiry: Storage-state sessions have finite lifetimes. If a platform expires cookies after 24 hours, the user must re-run the auth script. The UI shows session status so this is immediately visible.
  • DOM changes can break input/send selectors: While response extraction uses a resilient content-diff approach, the message input and send button still rely on selectors (textarea, button[aria-label="Send"]). These may need periodic updates. We centralize all selectors in a single SELECTORS map for easy maintenance.
  • Rate limiting: Sending messages too fast may trigger platform rate limits. We mitigate this with configurable delays (default: 3 seconds between messages).
  • Single-platform PoC: The current implementation includes a real adapter for Character.AI only. Other platforms would require implementing their own adapter (but the framework is ready).
  • WSL DNS workaround: Playwright's Chromium in WSL can't resolve DNS through the WSL proxy. We work around this with --host-resolver-rules flags. This is environment-specific and not needed on native Linux/macOS.

Truncation Bug (fixed)

An earlier version of the response extraction logic returned only the last paragraph of Character.AI responses instead of the full message. This caused responses in the database, UI, and exported JSON to appear truncated. The root cause was in waitForResponse(): it returned newBlocks[newBlocks.length - 1] instead of joining all new text blocks. This has been fixed — responses are now captured in full by joining all new DOM text blocks with paragraph breaks, and a stabilization wait ensures streaming responses are fully received before extraction.

4. Extending to Multiple Platforms

The adapter pattern makes multi-platform expansion straightforward:

To add a new platform (e.g., Replika):

  1. Create src/automation/adapters/replika.ts implementing PlatformAdapter
  2. Create scripts/auth-replika.ts for session initialization
  3. Register the adapter in runner.ts's getAdapter() switch
  4. Add the option to the UI's adapter dropdown
  5. Add npm run auth:replika to package.json

No changes needed to: database schema, run execution engine, logging system, transcript UI, artifact storage, or export functionality.

Other expansion vectors:

  • Mobile apps (iOS/Android): Replace Playwright with Appium in platform-specific adapters. The runner, logging, and evidence collection layer remain identical.
  • API-based platforms: For platforms with public or reverse-engineered APIs, adapters can make direct HTTP requests instead of browser automation — much faster and more reliable.
  • MITM proxy approach: For encrypted API traffic, a mitmproxy-based adapter could intercept and record API calls alongside browser automation for deeper analysis.

Running the PoC

# Clone and install
git clone https://github.com/rithwik0908/CompanionBench.git
cd CompanionBench
npm install
npx playwright install chromium

# Set up database
npm run db:migrate
npm run db:seed

# Initialize Character.AI session (one-time)
npm run auth:character-ai
# → Browser opens, log in manually, press Enter to save

# Start the app
npm run dev

# Open http://localhost:3000
# → New Run → Character.AI (Real) → paste a chat URL → Launch
